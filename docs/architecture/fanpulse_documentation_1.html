OpenAI API Role

What OpenAI GPT-4o actually does

The OpenAI API is the reasoning engine of the agent — not a wrapper. It drives four distinct, non-deterministic tasks across the pipeline.

ℹ️
Mock-first vs LLM-powered: The sports data APIs (TheSportsDB, API-Football, SerpAPI, Twilio) are mocked — they return deterministic test data so the app runs without external API keys. But the intelligence layer — understanding what the user follows, deciding what to look up, interpreting results, and composing the digest — is fully powered by OpenAI GPT-4o via openai>=1.0.0.
🧠
1 — Entity understanding
GPT-4o reads the onboarding message and extracts structured preferences: teams, athletes, sports, schedule frequency, phone number, and WhatsApp consent. It handles nicknames ("Lakers"), multi-word entities ("India Cricket"), and ambiguous references that a rule-based parser would miss entirely.
entity_extraction.py
🗺️
2 — Tooling plan
Before calling any tool, GPT-4o reasons over the user's preferences and decides which tools to call, in what order, and with what parameters. It produces a structured tool plan — e.g. call sports.fetch_fixtures for the Lakers, then athlete.get_ranking for Djokovic.
agent.py
📋
3 — Digest composition
GPT-4o receives the structured tool results and writes the digest — composing human-readable event cards, choosing what to emphasise, flagging unresolved entities, and assigning confidence scores. The output is WhatsApp-formatted markdown, not a template fill.
agent.py
❓
4 — Ambiguity resolution
When GPT-4o identifies an entity it cannot resolve with confidence (e.g. "India Cricket" could map to multiple tournament contexts), it generates clarification options and surfaces them to the user as inline choice buttons before proceeding.
agent.py
🔍
Tools: mock-first sports data
The five tool functions (fixtures, standings, news, rankings, WhatsApp send) return deterministic mock data by default. They are structured as real API stubs — live keys in .env switch them to real calls. GPT-4o interprets whatever data the tools return, mock or live.
tools.py — mock-first
✅
Human gates — LLM-unblocked
Three approval gates pause the GPT-4o pipeline and require explicit user action before the agent continues. No LLM call proceeds past a gate without human sign-off. The WhatsApp send is always mocked and always approval-gated — two independent guards.
agent.py — gates
Architecture

System architecture

FanPulse is a single-process Python application. The Streamlit UI, OpenAI-powered orchestration agent, MCP-style mock tools, and SQLite persistence all run in the same process.

System architecture — Orchestrator highlighted in blue. OpenAI GPT-4o powers the entire Agent layer. Sports API tools are mock-first (live stubs available). WhatsApp send is always mocked and always approval-gated.

Agent workflow

Agent state machine & edges

The agent moves through five GPT-4o-driven stages separated by three human approval gates. Every LLM call is non-deterministic — the agent adapts to what the user actually said, not a fixed template.

Agent state machine — Orchestrator swim lane (centre) powered by OpenAI GPT-4o throughout. Three HITL gates block autonomous write actions. Dashed lines = return/callback edges.

User journey

End-to-end flow

From a single onboarding message to an approved WhatsApp digest — four user interactions, three approval gates, and multiple GPT-4o calls.

1
User sends onboarding message
Free-text input. GPT-4o reads it via function calling and extracts structured preferences — teams, athletes, sports, delivery schedule, phone number, and WhatsApp consent. It handles nicknames, ambiguous entities, and natural phrasing that rule-based parsing would break on.
🤖 GPT-4o — entity extraction
2
Gate 1 — Ambiguity clarification (conditional)
If GPT-4o returns an entity with low confidence (e.g. "India Cricket" maps to multiple tournament contexts), the orchestrator surfaces three clarification choices as inline buttons. The user picks one; the resolved entity is stored and the extraction re-runs.
Human-in-the-loop · GPT-4o generated choices
3
Gate 2 — Confirm preferences
A structured preference card renders in the phone UI showing everything GPT-4o extracted. The user confirms before any tool is called or profile is written to SQLite. This is the last point to correct extraction errors.
Human-in-the-loop
4
GPT-4o plans and executes tool calls
The orchestrator passes the confirmed preferences to GPT-4o, which reasons over what to look up and produces a tool plan. It then calls sports.fetch_fixtures, sports.get_standings, web.search_news, and athlete.get_ranking — all returning mock data by default.
🤖 GPT-4o — tool planning + interpretation
5
GPT-4o composes the digest
GPT-4o receives all tool results and writes the digest — generating human-readable event cards, deciding what to highlight, flagging unresolved preferences, and assigning confidence scores. Output is WhatsApp-formatted markdown. This is non-deterministic: the exact phrasing adapts to the data.
🤖 GPT-4o — digest composition
6
Gate 3 — Approve digest before send
A full digest preview renders with all event cards, confidence scores, and flagged gaps. The user approves to send or requests changes (loops back to step 4). The WhatsApp send is always mocked — no real message is dispatched under any circumstances without explicit approval.
Human-in-the-loop · write action guard
7
Digest sent + trace logged
The WhatsApp message is generated and the mock send executes. The full agent trace, all GPT-4o calls, tool run log, and digest are written to SQLite. The Agent Trace / Debug View in the app exposes every step including the raw LLM reasoning.

Agentic design patterns

Key architectural decisions

⚙️
Orchestrator pattern
A single FanPulseAgent class is the sole orchestrator. It owns all state transitions, all GPT-4o calls, and is the only entry point for both the Streamlit UI and the scheduled job. No peer-to-peer agent communication — all intelligence flows through the orchestrator.
🤖
LLM-powered preference understanding
Entity extraction is done via GPT-4o function calling — not regex or keyword matching. This means the agent correctly handles "the Lakers", "Djokovic", "India cricket", mixed-language inputs, and preference corrections. The extraction is non-deterministic and context-aware.
🔌
MCP-style tool naming
All tools follow Model Context Protocol naming: namespace.action. GPT-4o reasons over tool names to decide which to call. Tool calls and results are logged to SQLite and visible in the debug trace view.
🧪
Mock-first tools, live AI
Sports data APIs are mocked so the app runs without credentials. But the intelligence layer — GPT-4o — is always live. This means you can demo the full agent reasoning loop without any sports API keys, while the LLM produces real, non-deterministic output.
🚦
Human-in-the-loop gates
Three explicit approval gates block write actions regardless of what GPT-4o produces. No profile write, no tool execution, and no WhatsApp send happens without user confirmation. The gates are implemented as required UI interactions — not prompt instructions.
🔄
Idempotent weekly job
The scheduled runner uses a run_key (ISO week number) to prevent duplicate digests. It skips users who already received a digest this period and logs per-user failures without stopping the batch. Fully decoupled from the Streamlit UI path.
Setup

Quick start

# 1. Clone and install (Python 3.11+)
git clone https://github.com/mnsrhz/FanPulse_Agent && cd FanPulse_Agent
python3.11 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. Add your OpenAI API key (required for LLM reasoning)
cp .env.example .env
echo "OPENAI_API_KEY=sk-..." >> .env

# 3. Run (sports APIs remain mocked — only OpenAI key needed)
PYTHONPATH=src streamlit run app.py

# 4. Paste sample onboarding:
"I am Mansoor. I follow the Lakers, Real Madrid, India cricket,
Novak Djokovic and Max Verstappen. Send my digest every Friday
morning to +14155550123 on WhatsApp."

# 5. Run scheduled weekly job
PYTHONPATH=src python weekly_digest_job.py

# 6. Tests
PYTHONPATH=src pytest tests/ -v
Roadmap

Future enhancements

🔑
Live sports API integrations
Swap mock tools for live TheSportsDB, API-Football, SerpAPI, and Twilio calls by adding keys to .env. No agent or LLM logic changes required — GPT-4o already interprets whatever the tools return.
🧩
Structured outputs
Migrate GPT-4o calls to use OpenAI structured outputs / JSON schema for more reliable extraction and digest formatting — reducing the need for post-processing and confidence scoring.
📅
Cron scheduling
Wire weekly_digest_job.py to GitHub Actions or a Render Cron job for fully automated weekly delivery without manual runs.
🌍
Multi-sport expansion
Add sport-specific tool modules for tennis (ATP/WTA), Formula 1, IPL, and NFL with deeper stats — GPT-4o will automatically plan calls to the new tools based on user preferences.
🔔
Event-triggered alerts
Add real-time push notifications (match starting soon, score updates) using the same WhatsApp tool layer with a reduced approval gate for low-risk alerts.
🧠
LangGraph migration
Refactor the orchestrator into a formal LangGraph state machine for cleaner edge definitions, LangSmith tracing, and checkpointed human-in-the-loop interrupt patterns.
